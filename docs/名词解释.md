# 名词解释

与本项目相关的核心术语简述，便于阅读 `实现阶段规划.md` 与实验报告。

---

## RAG（检索增强生成）

**Retrieval-Augmented Generation**：先查资料，再让大模型回答。

通用大模型不了解本校场馆细则，直接问易**编造**（幻觉）。RAG 流程为：

```text
用户问题 → 从知识库检索相关段落 → 与问题一并送入 Prompt → DeepSeek 生成回答
```

| 在本项目中 | 对应模块 |
|------------|----------|
| 知识库 | `knowledge/*.md` |
| 切块/索引 | `indexer.py`、`data/chunks.json` |
| 检索 | `retriever.py` |
| 拼 Prompt | `prompts.py` |
| 生成 | `llm.py`（DeepSeek） |

**要点**：RAG 是一套流程，不是单独模型；任务书要求回答有据、库外不编造。

---

## Embedding（文本向量化）

将文本编码为**数值向量**；语义相近的句子，向量距离更近。

用于**语义检索**：按相似度找段落，而非仅匹配关键词（如「羽球」≈「羽毛球」）。

```text
建库：每段资料 → 向量 → 存入 data/embeddings.npz
查询：用户问题 → 向量 → 与库中向量算相似度 → 取 Top-K 段供 RAG 使用
```

| 角色 | 说明 |
|------|------|
| Embedding | 负责「找哪几段资料」 |
| DeepSeek | 负责「根据资料写推荐」（任务书强制 API） |

检索可设**相似度阈值**：过低则视为无相关资料，配合边界处理防胡编。

---

## Gradio

Python 库，用少量代码搭建**网页聊天界面**，用于演示与答辩（任务书进阶加分；必做仅需命令行）。

```text
浏览器 ↔ app.py（Gradio）↔ conversation / retriever / llm（与 CLI 共用核心逻辑）
```

**要点**：`app.py` 只做 UI；业务逻辑与 `main.py` 共用，避免两套实现。

---

## 三者关系（简图）

```text
Gradio（界面）→ 多轮对话 → RAG 检索（常用 Embedding）→ DeepSeek 生成
                      ↑
                 knowledge 知识库
```

| 概念 | 任务书 | 本项目 |
|------|--------|--------|
| RAG | 必做 | 场馆资料检索 + 推荐 |
| Embedding | 进阶 | 语义检索（替代关键词） |
| Gradio | 进阶 | 网页聊天可视化 |

## 本地模型 vs 连 huggingface.co：效果差很多吗？

**不会。** 检索质量、回答内容几乎一样。

| 对比项 | 本地缓存加载 | 连官网 / 镜像下载 |
|--------|----------------|-------------------|
| 模型权重文件 | 同一份（已在 `C:\Users\ROG\.cache\huggingface\`） | 从网上拉同一份到缓存 |
| 推理结果 | 相同向量、相同相似度排序 | 相同 |
| 差别在哪 | **只省网络**：不 HEAD、不下载，启动更快、校园网不超时 | 首次下载或校验元数据时需要联网 |
| 何时必须联网 | 缓存缺失、版本要对齐、或 `EMBEDDING_LOCAL_ONLY=False` 补全 | VPN 或 `HF_ENDPOINT` 镜像 |

一句话：**本地 = 已经下好的同一套模型，只是不再问官网「有没有更新」**；不是换了一个「阉割版」模型。

---

## Hugging Face / huggingface.co 是什么？

**Hugging Face**（常写作 HF）是开源 AI 社区的「模型与数据集托管站」，类似程序员的 GitHub，但主要放**预训练模型**。

| 名词 | 含义 |
|------|------|
| **huggingface.co** | Hugging Face 的**官方网站域名**；浏览器、`huggingface_hub` 库默认从这里查模型信息、下载文件 |
| **Hub（模型中心）** | 网站上的模型仓库列表；每个模型有 ID，如 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` |
| **缓存目录** | 本机 `C:\Users\ROG\.cache\huggingface\hub\`，下载后存成 `models--sentence-transformers--...`，下次可离线读 |
| **HEAD 请求** | 库先向官网问「这个配置文件还在不在、版本对不对」；校园网常卡在这一步（你终端里的 `WinError 10060`），与权重是否已在本地无关 |

国内访问 `huggingface.co` 经常慢或被拦，所以项目里用 **`HF_ENDPOINT = https://hf-mirror.com`**（镜像站）：下载和校验走国内节点，**文件内容与官方一致**。

## sentence-transformers（语义向量库）

Python 库，把句子变成**固定长度向量**（本项目约 384 维），用于算「问题」和「知识库段落」有多像。

| 在本项目中 | 说明 |
|------------|------|
| 配置 | `config.py` 里 `EMBEDDING_BACKEND = "sentence_transformers"` |
| 代码 | `embedder.py` 的 `_embed_st`、`embed_query` |
| 默认模型 | `paraphrase-multilingual-MiniLM-L12-v2`（多语言，含中文） |
| 与 DeepSeek 关系 | **只做检索**，不替代大模型生成 |

**启动预热**：`embedder.warmup()` / `main.py` 启动时执行一次 `embed_query("预热")`，提前把模型载入内存，避免用户第一句等 30 秒～2 分钟。

---

## TF-IDF（离线备选向量化）

**Term Frequency–Inverse Document Frequency**：按字/词出现频率统计向量，**不下载神经网络模型**，完全离线、速度快。

| 对比 sentence-transformers | TF-IDF | 语义模型 |
|---------------------------|--------|----------|
| 理解同义词 | 弱（「羽球」「羽毛球」可能不像） | 强 |
| 校园网 | 友好 | 需先缓存模型或镜像 |
| 本项目 | `EMBEDDING_BACKEND = "tfidf"`，`data/tfidf_vectorizer.joblib` | 默认进阶方案 |

写报告时可对比两种后端；演示语义检索用 sentence-transformers 即可。

## 与本项目相关的其它术语

### 切块（Chunk）

把一篇 `knowledge/某馆.md` 按 `##` 标题拆成多段，每段一条记录写入 `data/chunks.json`。检索单位是**段**，不是整文件。

### 索引重建（`indexer.py --rebuild`）

重新扫描 `knowledge/` → 生成 `chunks.json` → 为每段算向量 → 写入 `embeddings.npz`。改知识库后需重建。

### 余弦相似度（Cosine Similarity）

两个向量夹角的度量，范围约 0～1（已归一化时）。`retriever.py` 用其排序；低于 `SIMILARITY_THRESHOLD` 视为**无命中**，Prompt 里要求模型不编造场馆。

### Top-K

检索只取相似度最高的 **K** 段（`TOP_K`）；拼进 Prompt 的场馆数还可再限（`MAX_VENUES_IN_PROMPT`），控制 token 与费用。

### `EMBEDDING_LOCAL_ONLY`

`True`：只读本机缓存，**不访问** huggingface.co（适合模型已下全的校园网日常运行）。  
`False`：允许联网/镜像补下载或校验；首次建索引或缓存坏了时临时使用。

### 多轮对话（本项目 P3）

`conversation.py` 保存 `history`，把上一轮用户话与本轮追问拼成检索 query（如「第二个怎么预约」），再调 RAG + DeepSeek。

### 边界处理 / 越界

`prompts.py` 用关键词判断问题是否超出「场馆预约」范围（如转专业、绩点）；越界则固定话术，不调场馆检索或限制编造。

### Token / API

DeepSeek 按**输入+输出字数（token）**计费。`TEMPERATURE` 越低回答越稳；RAG 塞太多段落会增加费用，故限制 `MAX_VENUES_IN_PROMPT`。

### DeepSeek API

任务书要求的大模型接口；`llm.py` 负责请求 `api.deepseek.com`，与 Embedding、Hugging Face **是两套独立服务**（一个生成文字，一个算向量）。

---

## 数据流（带名词对照）

```text
knowledge/*.md
    → indexer 切块 → chunks.json
    → embedder 向量化 → embeddings.npz（建库一次）

用户输入
    → conversation 多轮改写 query
    → retriever：embed_query（sentence-transformers 本地或 TF-IDF）
    → 余弦相似度 + 阈值 → Top-K 段落
    → prompts 拼 RAG 消息
    → llm DeepSeek 生成回答

Gradio（计划 P5）只换最外层界面，中间链路相同。
```
