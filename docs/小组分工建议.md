# 小组分工建议（3 人组）

> 适用：校园智能场馆匹配平台 · 方向一 RAG 项目  
> 原则：**每人都要参与编码与答辩**（任务书要求）；代码多的一方牵头，但其他人必须能讲清自己负责的模块。

---

## 1. 分工总原则

| 原则 | 说明 |
|------|------|
| 按模块拆，不按「谁只会写文档」拆 | 文档人人参与，但核心业务每人至少 1 个 `.py` 模块 |
| 经验多者 = 架构 + 联调 + Code Review | 定接口、合 PR、解决难点；不是包办全部代码 |
| 经验少者 = 独立模块 + 固定测试用例 | 任务边界清晰、可验收 |
| 共同资产 | `knowledge/` 资料收集、测试用例、实验报告、演示，**三人都要碰** |
| 答辩准备 | 每人准备 5 分钟：自己模块 + 在全流程图中的位置 |

---

## 2. 推荐角色（对应任务书）

任务书建议：设计/调优、开发/集成、文档/演示 —— 3 人时可 **一人主攻一角，并交叉备份**。

| 角色 | 主责 | 建议人选 |
|------|------|----------|
| **A · 技术负责人 / 集成** | 架构、模块接口、`llm.py`、联调、Git、帮队友 Review | 代码经验较多的同学（你） |
| **B · 数据与检索** | 知识库收集、`indexer.py`、`embedder.py`、`retriever.py` | 另一位（适合跟资料、测试打交道） |
| **C · 交互与质量** | `conversation.py`、`prompts.py`、`main.py`、`app.py`、测试用例、截图 | 另一位（逻辑相对直观、重体验） |

**交叉备份（防「一人请假全停摆」）**：

- A 熟悉 B 的检索流程；B 熟悉 C 的多轮与边界；C 能跑通 A 的 API 配置  
- 报告与答辩：**三人轮流写不同章节**，不是只让一人写

---

## 3. 按实现阶段的分工（与 `实现阶段规划.md` 对齐）

| 阶段 | 内容 | A（集成） | B（数据检索） | C（交互质量） |
|------|------|-----------|---------------|---------------|
| P0 | DeepSeek 配置、`llm.py` | **主导** | 录入 5+ 份 `knowledge/` | 整理资料模板、核对段落数 |
| P1 | `indexer.py`、切块 | Review | **主导** | 参与切块规则讨论 |
| P2 | 向量检索 | 协助阈值调参 | **主导** `embedder`+`retriever` | 写检索相关测试用例 T1/T4 |
| P3 | 多轮 + 边界 | Review Prompt | 提供「无检索」测试数据 | **主导** `conversation`+`prompts` |
| P4 | CLI 联调、README | **主导** 联调 | 跑检索回归测试 | **主导** `main.py`、测试记录 |
| P5 | Gradio | 部署/端口说明 | 知识库演示数据 | **主导** `app.py` |
| P6 | 报告、视频、答辩 | 报告「架构+RAG」章 | 报告「检索+对比实验」章 | 报告「交互+测试」章 |

## 4. 模块归属一览（答辩按模块讲）

```text
knowledge/          → B 主责，A/C 各录入一部分（每人 ≥5 段）
indexer.py          → B
embedder.py         → B
retriever.py        → B（A Review）
llm.py              → A
prompts.py          → C（A Review）
conversation.py     → C
main.py             → C
app.py              → C
config / README     → A 主笔，B/C 各写一节「如何运行」
docs/实验报告.md    → 三人分章节
docs/测试用例.md    → C 主笔，三人各补 2 条
docs/AI协作日志.md  → 每人记自己协作条目
```

## 5. 让「经验较少」的同学也能讲清代码

### 5.1 开工前（1 次 1～2 小时，三人一起）

1. 一起读：`README.md`、`名词解释.md`、`实现阶段规划.md` §2 架构图  
2. 你（A）用 15 分钟讲清数据流：**用户输入 → 检索 → Prompt → DeepSeek → 回复**  
3. 当场画一张白纸/白板流程图，**拍照放进 `docs/`**，作为组内共识  

### 5.2 开发中

| 做法 | 目的 |
|------|------|
| 模块接口先定函数名和入参（你写 10 行 stub） | 队友只需「填实现」，减少懵圈 |
| 每周 1 次 30 分钟「走读代码」 | 每人屏幕共享讲自己改的文件 5 分钟 |
| 每人至少 1 次「提 PR / 合并」到主分支 | 证明参与编码，非只挂名 |
| 固定测试句写进 `docs/测试用例.md` | 验收标准明确，不扯皮 |

### 5.3 答辩前（每人必会答的 3 题）

1. 用户一句话进来，**经过哪几个文件**？  
2. **RAG 在哪一步**？没有相关资料时程序怎么处理？  
3. **多轮对话** history 存在哪、怎么传给 DeepSeek？

答不上来的模块 → 当晚由负责人带着过一遍代码。

## 6. 经验较多同学（你）的时间分配建议

避免变成「只有你写代码」：

| 你做 | 尽量不做 |
|------|----------|
| 定目录、接口、第一次跑通 P0 | 包揽全部 P2–P5 实现 |
| 难点的 1 次示范（如向量检索） | 替写队友负责的整个模块 |
| 合并前 Review、答辩前总串讲 | 独自写整份实验报告 |

**推荐**：P0 你带跑通 → P1 你带 B 做 indexer 示范 → 之后 B/C 主模块你 Review。

| 成员 | 角色 | 具体工作 | 贡献占比 |
|------|------|----------|----------|
| 姓名 A | 技术集成 | llm、联调、README、架构… | __% |
| 姓名 B | 数据检索 | knowledge、indexer、embedder、retriever… | __% |
